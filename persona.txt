## PROJECT SPECIFICATION

### Overview
**GrihaNet** (गृहNet — Home Network) is a web-based smart home dashboard that unifies three household monitoring capabilities into a single platform:
1. **Power Monitoring** — Real-time energy tracking, appliance-level usage, cost calculation (₹/kWh), anomaly alerts, energy-saving recommendations
2. **Network Monitoring** — Device scanning, unknown device detection, bandwidth tracking, device blocking, speed test, screen time estimation
3. **Surveillance & Security** — Live camera feeds, OpenCV frame-differencing motion detection, snapshot logging, timestamped alerts, severity classification

### Societal Problem Addressed
Indian middle-class households face three converging problems that individually have partial solutions but lack a unified, affordable platform:
- **Energy waste**: 15-25% overpayment on electricity due to zero visibility into appliance-level consumption
- **Network insecurity**: 5-10+ connected devices per household with no parental controls or unknown device detection
- **Expensive surveillance**: Professional home security costs ₹15,000-50,000+; software-driven alternatives don't exist for average users

### Target Users
Primary: Middle-class Indian homeowners (25-55 age group) who manage household utilities, have 5-10 connected devices, and want affordable security. Secondary: Tech-savvy family members who configure the system.

## SDLC MODEL: AGILE (Iterative, Sprint-Based)

### Why Agile?
1. **Requirements evolved mid-project** — Parental controls feature was added in Week 3 after team discussion; Waterfall would have locked initial requirements
2. **Incremental delivery** — Each sprint produces a working increment (Sprint 1: backend scaffold → Sprint 2: core modules → Sprint 3: polish → Sprint 4: testing/docs)
3. **Continuous feedback** — Professor feedback after Task 2 influenced design decisions for Task 3
4. **Small team (4 members)** — Agile works best with small, co-located teams (Ref: Sommerville Ch.3)
5. **Academic constraints** — 9-week timeline requires rapid iteration, not sequential phases

### Why NOT other models?
- **Waterfall**: Too rigid; can't accommodate requirement changes; testing only at the end
- **Spiral**: Overhead of formal risk analysis at each spiral is excessive for a 9-week project
- **V-Model**: Assumes complete requirements upfront; testing phases are too heavyweight
- **RAD**: Requires experienced team with mature tools; academic team is still learning

### Sprint Timeline (4 Sprints, 9 Weeks)
- **Sprint 1 (Weeks 1-2)**: Planning, setup, database schema, API scaffold, UI wireframes
- **Sprint 2 (Weeks 3-5)**: Core module development — all 3 modules functional
- **Sprint 3 (Weeks 6-7)**: Polish — smart features, settings, responsive design, alert engine
- **Sprint 4 (Weeks 8-9)**: Testing, documentation, demo prep, final submission

**Ref**: Sommerville Ch.3 — Agile Software Development; Pressman Ch.3 — Agile Development

## DATABASE SCHEMA (7 Tables, 3NF Normalized)

```
USER (user_id PK, name, email, password_hash, role, created_at)
  │
  ├── APPLIANCE (appliance_id PK, name, icon, watts, room, is_on, user_id FK, created_at)
  │     └── POWER_READING (reading_id PK, appliance_id FK, watts, timestamp)
  │
  ├── NETWORK_DEVICE (device_id PK, name, ip, mac UK, type, bandwidth_used, is_online, is_whitelisted, is_blocked, user_id FK, last_seen)
  │     └── BANDWIDTH_LOG (log_id PK, device_id FK, download_mb, upload_mb, hour, date)
  │
  ├── CAMERA (camera_id PK, name, location, status, stream_url, user_id FK)
  │     └── MOTION_EVENT (event_id PK, camera_id FK, event_type, severity, snapshot_path, timestamp)
  │
  └── ALERT (alert_id PK, alert_type, message, icon, module, is_read, user_id FK, created_at)

SETTINGS (id PK, user_id FK, key, value)
```

### Normalization Justification:
- **1NF**: All attributes are atomic (no multi-valued attributes)
- **2NF**: No partial dependencies (all non-key attributes depend on the full primary key)
- **3NF**: No transitive dependencies (e.g., room is stored in APPLIANCE, not derived from another non-key attribute)

## DESIGN PRINCIPLES (Map each to GrihaNet code)

### SOLID Principles
| Principle | GrihaNet Implementation |
|-----------|----------------------|
| **S — Single Responsibility** | `routes/power.py` handles ONLY power logic; `routes/network.py` handles ONLY network logic. Each route file has one reason to change. |
| **O — Open/Closed** | Alert engine accepts new alert types (danger, warning, info, success) without modifying existing alert processing code. New modules can register alerts via `POST /api/alerts/create`. |
| **L — Liskov Substitution** | Camera base interface works with USB cameras, IP cameras, and ESP32-CAM. Any camera type can be substituted without breaking the surveillance module. |
| **I — Interface Segregation** | PowerChart React component receives only power-related props, not camera data. NetworkDeviceList doesn't receive power props. Components consume only what they need. |
| **D — Dependency Inversion** | Service layer depends on abstract database interface (SQLAlchemy ORM), not raw SQL. Switching from SQLite to PostgreSQL requires zero code changes in routes/services. |

### Other Design Principles
| Principle | GrihaNet Implementation |
|-----------|----------------------|
| **DRY** | Reusable Card, Stat, Badge, Toggle, ProgressBar components shared across all tabs |
| **KISS** | Standard REST GET/POST/PUT/DELETE, JSON responses, no GraphQL or over-engineering |
| **Separation of Concerns** | React = UI rendering only, Flask = business logic only, SQLite = data persistence only |
| **Modularity** | 3 independent modules; power module works even if network module is disabled |
| **High Cohesion** | All power-related logic in `routes/power.py` + `services/simulation.py` power functions |
| **Low Coupling** | Network module has ZERO import of power module. Only shared dependency = Alert Engine |

### Cohesion & Coupling Metrics
- **Cohesion Type**: Functional cohesion (highest level) — each module performs exactly one well-defined function
- **Coupling Type**: Data coupling (lowest practical level) — modules share only necessary data through the Alert Engine API
- **Cyclomatic Complexity Target**: < 10 per function (measured via `radon` tool)
- **Module Interdependency**: Only through the shared Alert Engine; no direct imports between power, network, and camera modules

**Ref**: Pressman Ch.8 — Design Concepts; Sommerville Ch.7 — Design and Implementation
